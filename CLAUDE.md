# KB1 — Enterprise Agent Knowledge Base

## Project Overview

Knowledge base compilation and retrieval system for automotive standards documents (e.g., QC/T 1036-2016). Converts raw PDF documents into structured facts, entities, and retrieval indices, then serves answer queries against the compiled knowledge.

## Architecture

```
raw PDF → pages/blocks → evidence → facts → entities/wiki/graph → retrieval → answer
```

Key modules:
- `query_rewrite.py` — Query type detection (definition/constraint/parameter_lookup/standard_lookup/comparison) and normalization
- `query_api.py` — Retrieval orchestration, FTS5 search, direct requirement injection
- `answer_api.py` — Answer generation, fallback chains, fact supplementation (~3200 lines, high cohesion)
- `evidence_judge.py` — Evidence sufficiency judgement (rule-based + optional LLM)
- `answer_policy.py` — Parameter answer building, efficiency/threshold handling
- `reranker.py` — Candidate re-ranking
- `retrieval_router.py` — Retrieval strategy selection

## Data Layout

```
knowledge_base/
  db/knowledge.db    — SQLite with FTS5 (facts, entities, wiki, graph tables)
  facts.db           — Compiled facts database (generated by build pipeline)
  documents/         — Source PDFs
  pages/             — Extracted page data
  blocks/            — Page block structure
  evidence/          — Evidence records
```

## Known Constraints

- **FTS5 uses `unicode61` tokenizer** — splits Chinese into individual characters, not words. CJK queries use 2-char fragment LIKE search as fallback.
- **LLM calls via httpx** — no openai/tiktoken SDK dependency. Direct HTTP API calls.
- **`answer_api.py` is a large file (~3200 lines)** — internal functions are highly interdependent; extraction requires care.
- **Test data dependency** — Tests referencing `18487.1.pdf` or `V2G.pdf` are auto-skipped when those documents aren't in the KB.

## Build Pipeline

```bash
# Compile knowledge base from documents
python -m enterprise_agent_kb.compile --workspace knowledge_base

# Run constraint regression script
python scripts/run_constraint_regression.py
```

## Query Type Detection

Query routing follows this priority:
1. Constraint keyword pre-check (保护重启, 过压保护, etc.)
2. `_has_explicit_constraint_intent` regex
3. `_has_explicit_parameter_lookup_intent` regex
4. LLM semantic parser (fallback, can be overridden by rule-based detection)

## Important Functions

| Function | Purpose |
|---|---|
| `_inject_direct_requirement_hits` | CJK fragment search in requirement/threshold facts |
| `_supplement_parameter_facts` | Adds requirement/threshold facts to parameter queries |
| `_strip_constraint_intent_suffix` | Strips intent markers from constraint queries |
| `_has_explicit_constraint_intent` | Regex-based constraint intent detection |
| `judge_evidence` | Rule-based evidence sufficiency check |
