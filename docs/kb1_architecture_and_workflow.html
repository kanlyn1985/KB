01

项目定位与核心理念

Positioning

KB1 是一个面向汽车行业标准文档（QC/T 1036-2016、GB/T 18487 系列、ISO 14229 等）的企业知识库编译与检索系统。它把原始 PDF 标准文档编译成可追溯、可证据约束、可回归保护的结构化知识库，再对外提供查询与答案服务。

核心数据流
编译期: raw PDF→ pages/blocks→ evidence→ facts→ entities/wiki/graph→ coverage/golden
运行期: retrieval→ answer→ evidence-constrained response

证据约束

答案只来自候选证据/事实/上下文。evidence_judge 是唯一事实裁决边界；LLM 不决定事实；证据不足必须降级而非补充。

可追溯

每条 fact → evidence → block → page → document 全链路溯源。source_units 是入库覆盖最小追踪单元，golden suite 提供回归保护。

闭环治理

六个闭环（入库/解析/召回/答案/回归/派生状态）持续度量与修复，证明系统"越改越稳"而非"看起来能跑"。

设计原则

单节点优先，不引入分布式依赖；偏好的是增量 schema 迁移而非破坏性重置；跨每个处理阶段保留质量元数据；数据模型可从 evidence 追溯到 facts 再到 wiki。

整体架构总览

Architecture Overview

系统分为三大层：编译层（把 PDF 变成结构化知识库）、查询层（召回 + 证据裁判 + 答案组装）、治理层（六闭环 + 派生状态 + 评测）。所有数据落在一个主 SQLite 库 knowledge_base/db/knowledge.db，外加只读本体库 ontology.db。

┌─ 编译层（CLI: build-document / build-batch）
register→ parse (PDF→pages→blocks)→ quality→ evidence→ facts→ entities→ wiki→ graph→ coverage→ acceptance→ post_ingestion_gate→ ontology terms

┌─ 治理层（CLI: workspace-doctor / eval / prune-stale-runs）
six loops dashboard→ derived-state governance→ golden suite + eval→ repair tasks

03

六闭环架构

Six Closed Loops

KB1 用六个闭环度量并修复系统。每个闭环有明确的关键对象、指标和修复动作，全部汇总到 /closed-loop-dashboard。这是"系统越改越稳"的核心机制。

LOOP 1

入库闭环 · Ingestion-Coverage

P0 已落地

职责：证明文档真的进来了，并定位未覆盖内容。最小追踪单元：source_units（4266 个），通过 source_unit_fact_map / source_unit_evidence_map 映射到 facts/evidence。验收入口：validate-document-ingestion。

指标：source_unit_coverage_rate 0.988、evidence_coverage_rate 1.0、fact_coverage_rate 1.0、uncovered_units 26（actionable 0，仅 test_gap_rejected）。

LOOP 2

解析质量闭环 · Parse-Quality

P1 已落地

职责：把解析风险拆成可处理根因，避免低可读性页面被误当入库失败。PDF provider 路由用 fast-text-first（_profile_pdf_text_layer 统计覆盖率/可读性），不达标回退 minimax+astron+paddle 慢路径。多解析视图 parse_views + page_parse_selection 选最可靠视图。

指标：parse_risk_pages 120、actionable_parse_risk_pages 0。归因类型：provider_quality_issue / selection_rule_issue / extraction_chain_issue / review_only / test_coverage_gap。

LOOP 3

召回闭环 · Retrieval-Quality

P1 已落地

职责：证明用户问法能找到正确内容并解释召回失败。召回路径：query_rewrite -> expansion -> topic_resolution -> graph_candidates -> retrieval_router -> rerank。retrieval_runs.code_version 是版本边界。

指标：Recall@5/10、MRR、nDCG@5、negative_hit_rate、must_hit_coverage、graph_retention_rate。Graph 是候选增强通道非事实裁决。

LOOP 4

答案闭环 · Evidence-Constrained-Answer

P1 已落地

职责：让答案只基于候选证据输出，歧义先澄清，不让 LLM 直接决定事实。evidence shape contract -> evidence_judge -> answer_policy -> direct answer。evidence_judge.sufficient=false 是硬门（sufficient=False AND conf<0.2 触发硬降级）。

指标：answer_pass_rate、citation_correct_rate、forbidden_contains_rate、answer_mode_accuracy、confidence_calibration。

LOOP 5

回归闭环 · Regression-Governance

P0 已落地

职责：用 golden suite（1466 case）+ corpus eval + 失败归因证明系统越改越稳。eval_runs.code_version 是版本边界；历史 repair_tasks 只作 backlog。Golden 候选生成有 readiness/activation gate，LLM 不直接决定。

指标：new_failures、fixed_failures、stable_pass_rate、retrieval/answer_regression_count。

LOOP 6

派生状态治理闭环 · Derived-State-Governance

P1 已落地

职责：发现、刷新、隔离过期派生数据（FTS 索引、retrieval/eval runs、孤立 DB 文件）。DerivedStateSpec registry + check_derived_state 返回 fresh/stale/missing。全部 dry-run 默认，严格安全边界（不删 golden/repair/facts/evidence/wiki/graph）。

工具：workspace-doctor（只读诊断）、workspace-governance（策略编排）、rebuild-derived-state（显式修复）、prune-stale-runs（运行派生物治理）、quarantine-suspicious-db-files（可疑 DB 隔离）。

闭环依赖关系

入库闭环是所有闭环的前置基础；回归闭环依赖召回+答案闭环产出；本体层是长期目标，依赖所有 P1 闭环稳定。解析依赖入库；召回依赖解析；答案依赖召回。

文档入库 Pipeline（13 阶段）

Ingestion Pipeline

文档编译由 pipeline._run_document_pipeline_unprotected 编排，按顺序执行 13 个阶段。每阶段通过统一 PipelineEvent 上报进度（CLI --progress 和 API background job 消费同一套事件）。任一阶段失败即中断，最后一条 stage event 是定位根因的第一证据。

3qualityquality_reports + parse_risk_profile
4evidenceevidence 表 + evidence_fts
5factsfacts 表（fact_type 分类）+ fact_evidence_map

06

查询与答案链路

Query & Answer Chain

查询链路从用户问题到可解释答案，全程遵守 LLM 边界：LLM 可扩写检索计划但保留硬锚点；Evidence Judge 只在候选 ID 集合内裁判；Answer API 不把 LLM 输出当最终事实。

user query -> 短缩写歧义? yes -> clarification_required -> no -> query_rewrite -> query_expansion (规则准入门 + fallback) -> advanced_query_planner (默认关闭) -> topic_resolution
topic_resolution -> graph_retrieval (候选增强) + -> retrieval_router (FTS + semantic + embedding) -> BP sibling expansion (topN 前) -> rerank (rerank_explanations) -> query context + retrieval_runs
context -> evidence shape contract -> evidence_judge (sufficient/confidence) -> answer_policy -> direct_answer + supporting_facts/evidence + ontology post_check (guard)

关键入口

入口作用
query_api.build_query_context构建结构化召回上下文；短缩写歧义先返回 clarification（不写 retrieval_runs）
answer_api.answer_query构建可解释答案；含 ontology post_check 与 warnings
query_expansion.expand_query结构化扩写，带规则准入门和 fallback
evidence_judge.judge_evidence按 evidence shape + 候选约束判定证据是否足够

answer_api 模块化（17 个模块）

原 ~3200 行 answer_api.py 已拆为：answer_policy、answer_fact_ranking、answer_process、answer_definition、answer_subgraph、answer_standard、answer_parameter、answer_comparison、answer_constraint、answer_context_routing、answer_evidence_selection、answer_query_parsing、answer_quality、answer_feedback、answer_utils + api_server/ 子包。

LLM 边界（硬约束）

① Query Expansion 可扩写检索计划但必须保留硬锚点；② Advanced Query Planner 默认关闭；③ Evidence Judge 只在候选 fact/evidence ID 集合内裁判；④ Answer API 不把 LLM 输出当最终事实来源；⑤ 短缩写定义查询先歧义澄清/规则回退，不先交 LLM 扩写；⑥ Graph 是候选增强非事实裁决。

已知边界规则

标准号查询（GB/T/GBT/GB/ISO/IEC）必须先解析 standard/document 实体，不沿 has_process 扩散；生命周期 BP sibling 扩展须在 topN 截断前；参数召回须同时满足对象锚点 + 物理量意图；定义查询术语召回：短缩写走 acronym 分支，无缩写才走中文长术语分支（Sprint 2 修复）。

07

Ontology 约束层

Ontology Layer

KB1 有两个相互独立但已最小接入的本体系统：主链路 ontology_adapter（约束层，ADR 0003）和独立 kb1_ontology（元域层，8041 LOC）。

ontology_adapter（主链路约束层）

连接只读 ontology.db（3.6MB，13 表：class_def/entity/term/relation/attribute 等）。三模式 OntologyMode：off（默认，零开销）/ shadow（只读信号）/ guard（post_check 警告）。

产出：OntologySignal（query_entities、relation_checks、post_checks、changed_retrieval=False、changed_answer=False）。已在 build_query_context 和 answer_api._compose_final_answer 两处接入。

kb1_ontology（独立元域层）

8041 LOC，phase 0-6 全完成。子包：class_registry / entity_manager / relation_registry / attribute_store。含 ADR 0001（元域层规则）、ADR 0002（两级归一化）。

当前状态：结构正确但功能受限——364 个 entity 共享单类 CLS-OBC-STANDARD，relation_checks 恒空，guard post_check 从不告警。更丰富信号需 ontology.db 增加类多样性。

ADR 0003：本体是约束/校验层 only（永久约束）

① 不生成事实；② 不绕过 evidence_judge；③ Sprint 2 不改答案（answer_changed_by_ontology=false，Sprint 3 起需独立 ADR 才可放松）；④ adapter 内无 LLM；⑤ 不引入 OWL/RDF 或重型图数据库；⑥ ontology.db 不作主事实存储。约束 1/2/4/6 永久，约束 3 可经独立 ADR 放松。

反 metric-gaming 机制

安全诊断（WP4，只读）

answer_safety.diagnose_answer_safety 计算 citation_correct / unsupported_claim / title_block_citation / degraded_answer 四个 flag，集成进 EvalResult 的 safety_metrics。纯 Python、确定性、不调 LLM、不写 DB、不改答案。

LLM 与 Provider 边界

LLM Boundary

KB1 区分两套独立的 provider 路由：文本 LLM（统一单 provider）和 PDF 解析 OCR（多 provider）。两者互不影响。

文本 LLM（统一 provider）

get_text_llm_settings() 读 ANTHROPIC_BASE_URL（api_base）/ ANTHROPIC_AUTH_TOKEN 或 ANTHROPIC_API_KEY（key）/ TEXT_LLM_MODEL 或 CLAUDE_MODEL（默认 claude-3-5-sonnet）。

单 provider 契约：已移除 minimax->astron 两段 fallback。本地 Anthropic 兼容代理识别 Claude 模型名（路由到 GLM），拒绝厂商名如 astron-code-latest。详见 docs/dev/llm-provider-policy.md。

PDF 解析 OCR（多 provider）

parse._profile_pdf_text_layer 统计文本覆盖率/可读性/符号率。fast-text-first：覆盖率+可读性同时达标才用 pymupdf_fast_text；否则回退 minimax_primary + astron_backup + paddlevl 慢路径。

PyMuPDF fast text 非"裸文本直通"--会按章节号/附录号/步骤号拆 heading/paragraph blocks，保证 procedure/requirement/section facts 可从数字 PDF 形成独立知识单元。

eval 性能开关（环境变量）

环境变量作用默认
EAKB_DISABLE_EMBEDDING_SEARCH=1跳过 SentenceTransformer embedding 搜索（每次调用重载模型，慢）off（启用）
