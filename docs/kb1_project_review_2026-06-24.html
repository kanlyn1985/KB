执行摘要

Executive Summary

02

项目概览

What & Why

定位

KB1 解决"如何让 AI Agent 可靠地回答汽车标准文档问题"这一问题。核心挑战在于： 标准文档结构复杂（章节/表格/术语/过程）、CJK 全文检索困难、且答案必须可追溯、可解释、不可臆造。 系统选择了一条证据约束（evidence-constrained）路线，而非端到端 RAG/LLM 自由生成。

语料规模：当前知识库收录 16 份标准文档，含 1668 页 / 31260 个 block / 29988 条 evidence / 7629 条 fact / 1047 个实体 / 1204 条图谱边。

关键事实

项目周期2026-02-25 → 2026-06-03（约 3.5 个月）
提交总数65 次（5月 31 · 6月 21，feature 颗粒度）
运行依赖httpx + PyMuPDF（无 openai/tiktoken，直连 HTTP）
存储SQLite + FTS5（unicode61 分词，CJK 走 2-gram LIKE 回退）
主库体积272 MB（knowledge_base/db/knowledge.db）
健康检查10/10 PASS

核心数据流

从原始 PDF 到可回答查询，系统经过一条多阶段编译管线，每个阶段都保留可追溯产物：

设计哲学：证据约束

系统的核心安全边界是 Answer API 不允许把 LLM 输出当最终事实来源。Evidence Judge 只能在候选 fact/evidence ID 集合内裁判；当证据不足时答案硬降级为"证据不足以确认"，而非由 LLM 补全。 这是 KB1 区别于普通 RAG 的关键工程取舍。

03

系统架构：六闭环

Six Closed-Loop Architecture

系统由 六个控制闭环构成。每个闭环都有明确的输入产物、关键对象、健康指标，以及可观测的 dashboard 入口（/closed-loop-dashboard）。这六个闭环不是孤立的 dashboard 指标， 而是带有根因归因和修复动作的反馈回路。

LOOP 1

入库闭环 · Ingestion Coverage

已落地

document → pages → blocks → evidence → facts → source_units → coverage。以 source_units 为最小追踪单元， 通过 validate-document-ingestion 验收新文档，保证"文档真的进来了并能定位未覆盖内容"。

source_unit_coverage_rate 98.8% evidence_coverage_rate 100% fact_coverage_rate 100% uncovered_units 26（仅 test_gap_rejected）
LOOP 2

解析质量闭环 · Parse Quality

已落地

多解析视图（PDF 文本层 / HTML / OCR）按页 selection，并把解析风险按证据链归因为 no_evidence / evidence_without_source_unit / source_unit_without_fact / fully_backed， 避免低可读性页面被误判为入库失败。

parse_risk_pages 120 actionable_parse_risk_pages 0 归因类型 5 类
LOOP 3

召回闭环 · Retrieval Quality

已落地

query_rewrite → expansion → topic_resolution → graph_retrieval → retrieval_router → reranker → context。 Graph 作为候选增强通道（非事实裁决层），graph 贡献度以是否进入 top context 为准。 retrieval_runs.code_version 提供版本边界。

指标 Recall@5/@10 · MRR · nDCG@5 graph_retention_rate negative_hit_rate
LOOP 4

答案闭环 · Evidence-Constrained Answer

已落地

context → evidence shape contract → evidence_judge → answer_policy → direct answer。 短缩写歧义先走 clarification；标准定义类查询已启用"证据不足即硬降级"门禁，正扩展到 lifecycle/test_method/timing/parameter。

指标 answer_pass_rate · citation_correct_rate forbidden_contains_rate confidence_calibration
LOOP 5

回归闭环 · Regression Governance

已落地 · 待激活

golden suite + corpus eval → eval_run → failure analysis → repair_tasks → golden generation activation gate。 当前健康只看最新 eval run 的失败，历史失败作为 backlog，避免旧版本污染当前结论。 corpus scale eval 尚未大规模激活（eval 主线正在重设计，见 §7）。

指标 new_failures · stable_pass_rate retrieval/answer_regression_count golden candidate activation gate
LOOP 6

派生状态治理闭环 · Derived-State Governance

已落地

workspace-doctor（只读诊断）→ workspace-governance（策略编排）→ rebuild-derived-state（显式修复） → prune-stale-runs（运行治理）。负责 FTS/graph/wiki/coverage 派生物的 stale 检测与幂等重建， 配合 code_version 版本边界隔离历史噪声。

DerivedStateSpec registry --execute-safe 保守自动修 prune-stale-runs dry-run 默认
