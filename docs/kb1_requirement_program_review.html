2. 子系统全景

Requirement Resolver 是面向客户/项目维度的 OBC/DCDC 需求治理子系统，叠加在 KB1 之上，提供从需求包导入到 ECO 闭环的完整治理管线。它不替代 KB1 的证据约束答案链，而是作为独立的需求维度治理层，仅在显式启用时通过软路由回答需求类查询。

完整治理管线

客户/项目需求包导入 (import-package)
    ↓
候选抽取 (extract-candidates)        ← 规则优先半自动，仅候选不直接生效
    ↓
人工评审/提升 (review → promote-candidate)
    ↓
客户/项目 Profile（含继承） (profile inheritance)
    ↓
有效需求解析 (resolve / resolve-project)   ← 确定性解析：standard → baseline → customer → project_overlay
    ↓
差异与冲突扫描 (diff / scan-conflicts)
    ↓
合规矩阵 (compliance)
    ↓
影响分析 (impact)
    ↓
审批治理 (approval: request → approve → reject)
    ↓
基线冻结 (freeze-baseline)
    ↓
基线版本对比与漂移检测 (diff-baselines / baseline-drift)
    ↓
DV/PV/SOP 发布门禁 (release-gate)
    ↓
ECO 工程变更闭环 (create-eco → submit → approve → apply → close)
    ↓
自动化验证报告 (run-eco-cycle)

解析路径示例（确定性）

以"客户A P1项目 DCDC 输出纹波"为例，解析链如下（全部有证据绑定，不让 LLM 决定值）：

standard_mandatory   PROFILE-STD-DCDC-MANDATORY   DCDC 输出纹波 ≤ 80mV   (base_constraint)
product_family_baseline  PROFILE-DCDC-BASELINE    DCDC 输出纹波 ≤ 50mV   (tighten)
customer_common      PROFILE-CUST-A-DCDC-COMMON   客户A DCDC 输出纹波 ≤ 30mV (tighten)
project_overlay      PROFILE-CUST-A-P1            P1 DCDC 输出纹波 ≤ 30mV @ 85℃ (clarify)
→ 最终有效需求：≤ 30mV @ 85℃  冲突状态：none  证据状态：verified

4. 三个集成点与爆炸半径

Requirement Resolver 通过三个集成点接入 KB1，每个都设计为默认关闭、opt-in、可回退。

集成点 1：CLI 子命令族（cli/_requirement.py）

KB1 的 CLI 是模块化子命令族架构（每个域 _workspace/_build/_eval/_test/_serve 暴露 register_subcommand + handle_command）。新建 cli/_requirement.py 适配器遵循同一模式，在 _orchestrator.py 注册。

# cli/_requirement.py（32 行适配器，非侵入）
def register_subcommand(subparsers):
    parser = subparsers.add_parser("requirement", help="...")
    _configure_requirement_parser(parser)

def handle_command(args, schema_path) -> bool:
    if args.command != "requirement":
        return False
    result = handle_requirement_command(args.root, args)
    print(json.dumps(result, indent=2, ensure_ascii=False))
    return True

爆炸半径：仅在 _orchestrator.py 新增 1 行 import + 1 行 register + 1 行 handle 调度（共 +3 行）。不修改任何既有子命令族。

集成点 2：answer_api 软路由（默认关闭）

在 answer_query 入口、docstring 之后插入软路由守卫。环境变量 EAKB_ENABLE_REQUIREMENT_ROUTER=1 才启用，默认返回 None（既有答案链完全不受影响）。

def answer_query(workspace_root, query, limit=8, preferred_doc_id=None):
    """Public entrypoint: ..."""
    # requirement_router_mvp: opt-in soft route (default off)
    requirement_answer = try_answer_requirement_query(workspace_root, query)
    if requirement_answer is not None:
        return requirement_answer
    # ... 既有答案链不变 ...

软路由仅在以下条件全部满足时介入：①环境变量启用；②requirement schema 存在；③意图属于 requirement_effective / requirement_diff / requirement_conflict_scan；④planner confidence ≥ 0.75。否则返回 None，既有答案链接管。

爆炸半径：仅 +6 行（1 import + 5 行守卫），默认关闭。

集成点 3：api_server FastAPI 适配（可选）

提供框架中性 handler + 可选 FastAPI router。集成脚本仅在识别到安全 FastAPI 模式时自动 patch，否则拒绝并提示手工介入。

爆炸半径：仅当 api_server 识别为 FastAPI 模式才生效；当前 KB1 api_server 形态需手工确认，不会自动改动。

实测验证：软路由关闭时 try_answer_requirement_query 返回 None（既有链路零影响）；启用时返回 answer_mode=requirement_effective 的正确需求答案。definition 查询、CP 缩写查询、既有 eval 基线均无回归。

6. KB1 既有边界兼容性

Requirement Resolver 的设计与 KB1 的核心不变量完全兼容。逐项核对如下：

KB1 不变量Requirement Resolver 行为合规
evidence_judge 是唯一事实裁决边界解析器确定性，不让 LLM 决定有效需求值；证据绑定引用但不裁决合规
答案只能来自候选证据/事实/上下文软路由返回需求治理数据，非 KB1 evidence；仅 3 种需求意图介入合规
证据不足应降级而非补全confidence<0.75 时软路由不介入，回退到既有答案链（含降级）合规
不重写 query_api/answer_api 主链路answer_api 仅 +6 行软路由守卫，query_api 零改动合规
无新向量库/分布式服务使用 SQLite（同 KB1），无外部依赖合规
Ontology 输出只能是 signal/constraintRequirement Resolver 与 Ontology 独立，不交互N/A
answer_changed_by_ontology 必须 false软路由独立于 ontology post-check，互不影响合规
schema 变更应走 migrations/ 而非模块级 SQL当前 schema 在 requirements/schema.py（技术债，待迁移）技术债

7. 已知技术债

3 项技术债，均非阻塞合并，但应在合并后短期治理：

技术债 1：Schema 定义在模块级 SQL

requirements/schema.py 的 SCHEMA_SQL 常量定义了 28 张表。应迁移到 src/enterprise_agent_kb/migrations/0xx_requirement_program.sql，纳入 KB1 的 NNN_*.sql 迁移系统（apply_pending_migrations 按 version 号词法序应用）。

影响：当前 schema 由 requirement init-schema 命令触发，未纳入 bootstrap.initialize_workspace 的统一迁移路径。修复成本：低（拆 SQL 文件 + 注册 version）。

技术债 2：事务边界

ECO/基线/审批的跨表操作需要服务级事务收敛。当前各模块独立提交，长事务场景下可能出现部分提交。修复成本：中（需引入服务层事务封装）。

技术债 3：API 适配器双路径

api.py 同时提供框架中性 handler 和可选 FastAPI router。合并后应选择唯一集成路径，避免双路径维护负担。修复成本：低（确认 KB1 api_server 形态后删一路径）。

抽取阶段边界：extraction 是规则优先半自动候选生成，仅产出候选，绝不直接提升为有效需求。必须经 promote-candidate 人工评审。这是设计约束，非技术债。
