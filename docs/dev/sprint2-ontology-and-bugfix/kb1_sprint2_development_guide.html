给执行 AI 的总指令

Read First
你是被要求继续开发 KB1 的执行 AI。不要重新设计整个系统。

当前项目已经完成 Sprint 1 稳定化：工作树干净、测试全绿、WIP 已提交、eval baseline 已固定。你的任务是按照本文档顺序完成 Sprint 2，不要跳步，不要大重构，不要用“重新做一套 RAG / Ontology 系统”替代现有系统。

你必须优先保护成果

  • 确认当前分支与工作树状态。
  • 优先 push 或生成 git bundle 离线备份。
  • 任何代码改动前先跑 baseline。

你只能做小步改动

  • 每个工作包单独提交。
  • 每个提交都能解释业务目的。
  • 每个提交后至少跑相关测试。

你不能破坏证据边界

  • 不得让 Ontology 直接生成事实。
  • 不得绕过 evidence_judge。
  • 不得让 LLM 自由补全标准答案。

02

不可违反的系统约束

Non-negotiables

证据约束边界

  • evidence_judge 仍然是答案事实裁判边界。
  • 答案只能来自候选 evidence / fact / context。
  • 证据不足时仍然必须硬降级，而不是由 LLM 或 Ontology 补完。
  • Ontology 只能增强检索、约束、校验，不得成为事实来源。

工程治理边界

  • 保持 CodeStable 记录：issue / design / checklist / acceptance。
  • 不做破坏性 reset、不清空 DB、不重建不可追溯状态。
  • 所有新模式必须有开关，默认保守。
  • 所有行为变更必须有 regression test。
关键判断

Sprint 2 的核心不是“让本体系统很聪明”，而是让本体系统以低风险方式进入主链路，先成为 约束层 / 校验层 / 召回辅助层。不要让它越权成为 answer generator。

WP2

修复定义查询 bug，解除 MCP xfail

问题：definition-query-exact-term-gate-drops-evidence 表明定义类查询在精确术语门控中丢失 evidence，导致 MCP answer_query 可能回答成文档标题或证据不足。

目标：定义类问题必须稳定返回术语定义证据，而不是返回文档标题、章节名或空证据。

建议排查路径

  • 定位 xfail 对应测试：tests/test_mcp_server.py 或相关 answer query 测试。
  • 追踪 MCP tool → answer_api → query/retrieval → evidence_judge 的上下文。
  • 检查 definition query exact-term gate 是否过早过滤掉包含定义的 evidence。
  • 检查候选 context 是否把 document title 排在定义 evidence 前。

修复原则

  • 只修定义查询路径，不大改 answer 主链路。
  • 优先修候选选择 / gate 逻辑，不靠 prompt 让 LLM 猜。
  • 保留 evidence IDs，确保 citation 可追溯。
  • 补回归测试，防止以后再次答成文档标题。
# 修复后应执行
python -m pytest -q tests/test_mcp_server.py
python -m pytest -q -m "not integration and not benchmark"
验收标准
原 xfail 转为 pass；MCP answer_query 对定义问题返回定义证据；fast suite 仍 0 failed。
提交建议
fix(answer): preserve definition evidence through exact-term gate
WP3

Ontology 最小接入：只读影子模式

目的：让 kb1_ontology 进入主系统，但第一步只做只读信号，不改变答案结果。这个阶段的价值是验证接口、日志、指标和回归风险。

硬规则

Ontology 输出只能叫 signal / constraint / validation，不能叫 evidence / fact / source truth。不要把 ontology 推理结果写入最终答案事实。

# 建议新增/确认的配置模式
KB1_ONTOLOGY_MODE=off      # 默认：完全不影响主链路
KB1_ONTOLOGY_MODE=shadow   # 只记录信号，不改变排序/答案
KB1_ONTOLOGY_MODE=guard    # 仅在明确安全的 post-check 中提示风险，不生成事实
# 不允许 Sprint 2 直接启用 auto-answer / auto-fact 模式
接入点作用不得做什么
Entity Constraint识别查询涉及的实体类型，如 UDS 服务、DID、NRC、诊断会话、参数。不得直接新增答案事实。
Relation Domain/Range校验关系是否合理，如某 NRC 是否适用于某服务，某 DID 是否属于某上下文。不得覆盖 evidence_judge 裁决。
Answer Post-check对已生成答案做一致性检查，输出 warning / risk flag / validation report。不得把无证据的 ontology 推断写进答案正文。
验收标准
shadow 模式下，旧答案完全不变；日志/trace 中出现 ontology_signals；fast suite 仍全绿。
提交建议
feat(ontology): add read-only ontology signal adapter
WP4

Ontology 最小接入：Guard 模式与 Post-check

目的：在不改变事实来源的前提下，让 ontology 对答案做“结构一致性审计”。这一步适合解决标准知识库中常见的类型混淆、关系错配、参数范围不一致问题。

# 推荐输出结构，不要求字段名完全一致，但语义必须清楚
{
  "ontology_mode": "guard",
  "entity_constraints": [
    {"mention": "0x22", "class": "UDSService", "confidence": 0.95}
  ],
  "relation_checks": [
    {"relation": "applicable_to", "status": "consistent|unknown|conflict"}
  ],
  "answer_post_checks": [
    {"type": "unsupported_ontology_relation", "severity": "warning", "message": "..."}
  ],
  "changed_answer": false
}
  • 新增 post-check 测试：一致、未知、冲突三种情况。
  • 默认只返回 warning，不改变 answer text。
  • 只有当 evidence 本身支持时，才允许 answer policy 使用 ontology warning 改变置信度或提示。
验收标准
guard 模式输出可观测 post-check；默认答案不被 ontology 改写；无新增 unsupported claim。
提交建议
feat(ontology): add guarded answer consistency checks
WP5

Eval 提分：0.60 → 0.65–0.85

目的：在不换评分口径的前提下，提升 deterministic eval baseline。不要通过降低标准、换 metric、删题来“刷分”。

正确提分方式

  • 对失败样本做 bucket：召回失败、证据缺失、答案模板、citation 错误、expected point 绑定错误。
  • 优先修高频、低风险 bucket。
  • 完善 page-bound expected_points。
  • 修复定义/术语类问题后重新跑 eval。

错误提分方式

  • 切换到更宽松 metric。
  • 删除困难样本但不记录理由。
  • 让 LLM judge 自由判对。
  • 把 ontology 推断当证据补分。
# 每次修复后记录对比

05

建议技术方案：Ontology 接入架构

Minimal Safe Integration

推荐使用“主系统调用只读 adapter”的方式，不要让 kb1_ontology 反向依赖或控制主系统。主系统仍然掌握 query、retrieval、evidence_judge、answer_policy 的流程。

user query
  │
  ├─ query_semantic_parser / topic_resolution
  │
  ├─ ontology_adapter.analyze(query, candidates)   # shadow/guard only
  │      ├─ entity_constraints
  │      ├─ relation_domain_range_checks
  │      └─ post_check_rules
  │
  ├─ retrieval_router / reranker
  │
  ├─ evidence_judge                     # 事实裁决仍在这里
  │
  ├─ answer_policy
  │
  └─ ontology_post_check(answer, citations, evidence_ids)
         └─ 输出 validation / warning，不直接生成事实

推荐新增对象

OntologyMode = Literal["off", "shadow", "guard"]

@dataclass
class OntologySignal:
    query_entities: list[EntityConstraint]
    relation_checks: list[RelationCheck]
    post_checks: list[AnswerPostCheck]
    changed_retrieval: bool = False
    changed_answer: bool = False

推荐日志字段

retrieval_run.ontology_mode
retrieval_run.ontology_signal_count
answer_run.ontology_post_check_status
answer_run.ontology_conflict_count
answer_run.answer_changed_by_ontology  # Sprint 2 必须始终 false
