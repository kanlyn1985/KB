执行摘要

Read this first
本轮开发的唯一主线：把项目从“成果很多但状态发散”恢复为“可持续交付基线”。

不要马上推进复杂 ontology 推理，不要扩展新的 RAG 能力，不要重写 retrieval/answer 主链路。本轮必须先完成：WIP 分批入版本控制、8 个失败测试收口、评测 baseline 固定、工程垃圾清理、four-loop/six-loop 命名一致化。

应该做

  • 保存并提交当前 WIP，避免本地成果丢失。
  • 优先修复测试和契约漂移。
  • 把 eval 从“实验脚本”变成可复现门禁。
  • 本体层只做最小接入，不抢主系统控制权。

不要做

  • 不要破坏 evidence-constrained answer 边界。
  • 不要让 LLM 直接生成最终事实。
  • 不要 reset / 删除数据库 / 批量清空派生状态。
  • 不要未测试就批量改 retrieval、answer_policy、graph。

最终交付

  • 清晰 commit 序列。
  • pytest 结果和失败修复说明。
  • eval baseline 报告。
  • post_ingestion_gate 稳定报告。
  • 下一 Sprint 候选任务清单。
01

项目背景与当前状态

Context

KB1 是面向汽车标准文档的证据约束知识库系统。主链路是 PDF → pages → blocks → evidence → facts → entities/wiki → graph → retrieval → answer。系统的核心优势是：六闭环架构、CodeStable 治理、Evidence Judge 安全边界、CI 健康检查。

维度当前状态对下一步开发的含义
健康度B+；架构成熟但存在 WIP 和工程债务。先稳定，不要贸然扩展。
测试672 passed / 8 failed；失败多为 WIP、陈旧断言、契约变化。恢复 CI 绿灯是 Sprint 1 的硬门槛。
WIPevaluation/evaluator.py +722、pipeline.py +331、kb1_ontology 8K LOC 等未收口。先分批 commit/push，再做优化。
Evalpass_rate 在 0.13–0.70 间波动，Phase 1 尚未稳定进入 65–85%。固定主评分口径，形成可复现 baseline。
OntologyPhase 0–6 已做完，但仍是独立并行系统，部分未纳入版本控制。作为约束层接入，不要重写主知识库。
判断

当前最大风险不是代码质量差，而是“成果太多但没有被干净地纳入版本控制和 CI 体系”。这会导致后续任何 AI 接手时都难以判断哪些行为是设计变更、哪些只是半成品。

02

执行原则与硬约束

Rules

不可违反的系统边界

  • 答案事实来源必须来自 evidence/fact 候选集，LLM 不能越过 Evidence Judge 直接定事实。
  • 证据不足必须硬降级，输出“证据不足以确认”，不能为了好看而补全。
  • Graph 只能增强候选召回，不能作为无证据事实生成器。
  • Ontology 只能做约束、校验、有限推理，不能绕过 judge 生成新事实。
  • 禁止破坏性 reset：不得随意删除主库、清空 evidence/facts、重建不可恢复状态。

执行型 AI 工作纪律

  • 每次改动前先读相关测试和设计文档。
  • 每个 work package 单独 commit，不把 eval、ontology、cleanup 混在一起。
  • 改代码必须同时改测试、文档或报告。
  • 修复失败测试时先判断：是生产 bug、契约变更，还是陈旧断言。
  • 每一步输出命令、结果摘要、剩余风险。
优先级公式

保住成果 > 恢复绿灯 > 固定评测 > 小步接入 ontology > 新能力开发。 如果出现冲突，以左侧优先级为准。

03

Sprint 1 目标定义

Stabilization Sprint

目标 1

WIP 全部可追溯
所有已跟踪改动和 untracked 新系统完成分类、提交、推送或明确隔离。

目标 2

测试恢复可信
8 个失败测试修复或隔离为明确 WIP，不允许长期红灯。

目标 3

Eval baseline 固定
确定一个 CI 主口径，避免 token-overlap / strong-match / LLM judge 来回切换。

目标 4

Ontology 最小接入
本体层进入版本控制，并只接入 entity/relation/attribute 约束，不改主链路事实边界。

Sprint 1 不做事项

不做原因未来何时做
大规模重构 retrieval_router / query_api当前测试未绿，改动面会扩大。Eval baseline 稳定后。
完整 ontology 推理引擎容易与 evidence/fact 主链路争夺事实权。最小集成验证通过后。
新增大批标准文档当前入库和评测门禁未稳定。post_ingestion_gate 稳定后。
更换数据库/向量库/分布式依赖与项目轻量化边界冲突。除非有明确容量瓶颈证据。

06

Eval Baseline 固定方案

Evaluation Gate

当前 eval 最大问题不是“分数低”，而是评分口径切换导致无法比较。Sprint 1 的目标是先建立可复现 baseline，然后再提升分数。

推荐口径

  • CI 主口径：expected_points coverage + strong-match。
  • 辅助口径：LLM judge，仅作为人工分析，不阻断 CI。
  • 召回口径：Recall@5 / Recall@10 / citation correctness 单独记录。
  • 答案口径：unsupported claim rate 必须长期趋近 0。

Gate 分层

Gate含义
baseline lock分数不能相对当前固定版本明显退化。
promotion gatepass_rate 稳定进入 0.65–0.85。
release gate跨文档 eval、citation、unsupported claim 均达标。

需要生成的报告

docs/dev/sprint1-stabilization/eval_baseline_report.md
包含：
- run_id / code_version / db_version / qa_bank_version
- scoring_mode
- pass_rate / coverage_rate / citation_correct_rate / unsupported_claim_rate
- failed cases top buckets
- 是否允许进入下一阶段
重要判断

讯飞 LLM API 500 不能继续让整个 eval 主线卡死。执行 AI 应把 LLM judge 退到 optional，并保证没有外部 LLM 服务时仍能跑 deterministic eval。

07

Ontology 最小接入方案

Minimal Integration

本体层已经有 class_registry、entity_manager、relation_registry、attribute_store、combined_query.py、legacy_bridge.py 等模块。Sprint 1 只允许做最小接入，不允许改造成大推理系统。

接入点 1：Entity Type Constraint

把实体类型作为召回/答案前的过滤与校验信号。例如 UDS 服务、DID、NRC、诊断会话、测试条件、参数范围。

接入点 2：Relation Domain/Range

校验关系是否合理。例如 applicable_to、belongs_to、has_parameter 的主体/客体类型是否匹配。

接入点 3：Answer Post-check

回答生成后做语义一致性检查，但不能改写事实。发现冲突时降级或标记不确定。

禁止的 ontology 行为

  • 禁止 ontology 直接生成新的 fact 并进入答案。
  • 禁止 ontology 绕过 evidence_judge。
  • 禁止引入 OWL/RDF 或重型图数据库作为本阶段依赖。
  • 禁止把 ontology.db 作为主事实库替代 knowledge.db。
